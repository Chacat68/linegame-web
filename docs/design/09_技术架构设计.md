# 星际贸易商 - 技术架构设计

## 系统概述

本文档详细描述了《星际贸易商》的技术架构设计，包括代码结构、数据管理、性能优化和扩展性考虑。基于 Web（浏览器）运行时的特性，我们采用模块化、可扩展的架构设计，确保游戏的稳定性和可维护性。

## 整体架构

### 架构模式

1. **MVC架构模式**
   - Model：数据模型和业务逻辑
   - View：用户界面和显示层
   - Controller：用户输入处理和逻辑控制

2. **系统模块化设计（基于 ES Modules）**
   - 每个玩法域拆分为独立 `js/systems/*` 模块
   - UI 拆分为独立 `js/ui/*UI.js` 模块
   - 模块间通过 `GameManager` 编排与 `EventBus` 事件通信

3. **分层架构**
   - 表现层：`js/ui/*` + `Renderer`
   - 业务层：`js/systems/*` + `GameManager` 动作处理
   - 数据层：`js/data/*` 静态配置 + `SaveSystem` 持久化
   - 基础层：`EventBus`、浏览器 Canvas/WebGL/Storage API

### 核心模块

1. **游戏管理器（GameManager）**
   - 游戏状态管理
   - 场景切换控制
   - 全局事件调度
   - 系统初始化和清理

2. **事件系统（EventBus）**
   - 全局事件总线
   - 事件订阅和发布
   - 跨系统日志/提示/完成通知

3. **系统模块（Systems）**
   - 经济、贸易、随机事件、舰队、派系、研究
   - 任务、成就、教程、胜利、存档
   - 各系统由 `GameManager.init()` 统一初始化并共享 `_state`

4. **界面模块（UI）**
   - `MapUI / MarketUI / ShipUI / HUD / Modal / *UI`
   - 负责输入采集和显示，不直接持久化核心状态

## 核心系统架构

### 全系统设计细节（按当前仓库）

| 系统域 | 核心模块 | 关键输入 | 关键输出 | 设计修正 |
|-------|---------|---------|---------|---------|
| 贸易 | `TradeSystem.js` | 玩家买卖动作、当前市场价格 | 库存/资金变更、交易日志 | 统一经 `GameManager` 处理，避免 UI 直接改状态 |
| 经济 | `Economy.js` | 天数推进、系统供需参数 | 实时买卖价、税率与波动更新 | 保持参数可配置，避免硬编码扩散 |
| 事件 | `RandomEvent.js` | 航行/天数推进/触发条件 | 奖惩、生效状态、事件日志 | 高惩罚事件需保底阈值，避免新手阶段劝退 |
| 舰队 | `FleetSystem.js` | 舰船配置、派遣指令 | 舰队收益、维护费用、船只状态 | 自动派遣节拍统一由主循环驱动 |
| 派系 | `FactionSystem.js` | 交易与事件行为 | 关系值、价格修正、声望变化 | 关系影响需透明化展示到 UI |
| 研究 | `ResearchSystem.js` | 研究选择、时间推进 | 科技解锁、经济与事件加成 | 研究乘区顺序固定，减少预期偏差 |
| 任务 | `QuestSystem.js` | 玩家行为事件、阶段条件 | 任务进度、奖励发放 | 任务触发与教程节奏联动 |
| 成就 | `AchievementSystem.js` | 统计指标、关键里程碑 | 成就解锁、奖励与提示 | 条件判定集中在系统层，避免重复计算 |
| 教程 | `TutorialSystem.js` | 新手阶段动作、页签点击 | 教程步骤推进/跳过 | 教程状态持久化，支持重启恢复 |
| 胜利 | `VictorySystem.js` | 净资产/研究/任务等综合进度 | 胜利达成判定与展示 | 多路径共享统一进度展示逻辑 |
| 存档 | `SaveSystem.js` | 保存/读档/导入导出 | `SaveEnvelope` JSON、槽位元信息 | 当前以 localStorage 为主，IndexedDB 作为后续迁移选项 |

## 数据架构

### 数据存储策略

1. **配置数据**
   - 使用JSON格式存储静态配置
   - 支持热重载和动态更新
   - 版本控制和兼容性检查

2. **游戏存档**
   - 使用 `SaveEnvelope` JSON（`meta + data`）
   - `schemaVersion` + 迁移函数预留
   - localStorage 4 槽位（0 自动存档，1-3 手动）
   - 支持导入/导出 JSON

3. **运行态策略**
   - 运行态单一 `_state` 保存在 `GameManager`
   - 静态数据放在 `js/data/*`，由系统按需读取
   - UI 临时状态不进入持久化（如 `hoveredSystem`）

### 数据模型设计

1. **存档封装结构（示例）**
```js
{
  meta: {
    schemaVersion: 1,
    gameVersion: '0.2.0',
    slotId: 0,
    timestampMs: 0
  },
  data: { /* game state */ }
}
```

2. **状态约束**
```js
// 仅示意：运行态字段不落盘
const data = JSON.parse(JSON.stringify(state));
delete data.hoveredSystem;
```

## 技术选型说明

### Web（浏览器）技术优势

1. **开发与迭代效率**
   - 前端生态成熟，开发/调试工具链完善
   - 热更新/快速迭代便于原型验证
   - UI 与数据可视化实现成本较低

2. **性能特点**
   - Canvas/WebGL 在 2D 场景下可满足多数需求
   - 浏览器具备较好的 JIT 与图形加速能力
   - 可按需加载资源，降低首屏与内存压力

3. **跨平台支持**
   - 天然跨平台：Windows/macOS/Linux + 多数移动设备
   - 分发成本低：URL 即用，便于灰度与回滚
   - 可结合 PWA 支持离线与安装体验（可选）

### 技术栈选择

1. **编程语言**
   - JavaScript（ES Modules）：主要开发与运行语言

2. **数据格式**
   - JSON：配置文件
   - localStorage：本地持久化（当前存档实现）
   - 二进制（ArrayBuffer/DataView）：高性能序列化（如关卡）

3. **工具链**
   - Git：版本控制
   - CI/CD：自动化流程
   - 性能分析工具：浏览器 Performance 面板、Lighthouse
