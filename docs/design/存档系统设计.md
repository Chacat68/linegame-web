# 星际贸易商 - 存档系统设计（Web 版本）

## 系统概述

存档系统用于将玩家游戏进度持久化到本地，核心目标是：

- 数据完整性：保存/加载可校验、可回退
- 版本兼容：支持 schemaVersion 迁移
- 可用性：多槽位 + 自动存档 + 可选导出/导入

在 Web（浏览器）技术基线下：

- 默认存储介质：IndexedDB（结构化存储，容量相对充足）
- 可选：导出/导入 JSON 文件（用于备份、迁移浏览器/设备）

---

## 存档数据结构

### 结构约定（SaveEnvelope）

存档建议采用“信封 + 负载”的结构，便于扩展、迁移与校验：

```json
{
  "meta": {
    "schemaVersion": 1,
    "gameVersion": "0.1.0",
    "slotId": 1,
    "saveName": "我的商业帝国",
    "timestampMs": 1760000000000,
    "playTimeSeconds": 18543,
    "lastPlanet": "新地球",
    "difficulty": "standard",
    "isAutosave": false,
    "thumbnailDataUrl": "data:image/png;base64,..."
  },
  "data": {
    "player": {},
    "fleet": {},
    "galaxy": {},
    "quests": {},
    "settings": {},
    "economyState": {},
    "unlocks": {}
  }
}
```

说明：

- `schemaVersion`：存档结构版本（用于迁移），与 `gameVersion` 分离
- `thumbnailDataUrl`：可选（Canvas 截图后 `toDataURL()`），注意体积与写入频率

## 存档策略

### 存档槽位管理

槽位配置：

```
总槽位数：4
├── 1-3：手动存档槽位
└── 0：自动存档槽位（特殊）
```

Web 实现建议：

- IndexedDB 中以 `slotId` 作为主键
- UI 展示来自 `meta` 的摘要信息

### 自动存档机制

触发条件（可配置）：

- 时间触发：间隔 3/5/10 分钟
- 事件触发：到达星球、完成重要任务、飞船升级等

条件限制（避免破坏一致性）：

- 不在关键交互中（例如结算动画/交易确认弹窗）
- 不在加载/场景切换过程中

写入策略：

- 写入前读取旧记录并保存到 `backup` 字段（或单独的 `saves_backup` store）
- 写入失败时保留旧记录，提示用户并记录错误

### 增量保存（可选）

若存档体量较大或自动存档频繁，可引入“脏标记（dirty flags）”机制：

- 系统层维护变更点（如 `player.credits`、`fleet.ships[0]`）
- 自动存档仅写入增量（delta）或仅写入被标记的子树

简化方案（推荐先做）：

- MVP 阶段先做“全量保存 + 限频（throttle）”，待性能出现瓶颈再升级

## 数据安全与防篡改（可选）

在纯前端环境中，任何“加密密钥”都会随代码分发，因此存档安全更多是：

- 防误操作与防损坏（校验、备份、回退）
- 防低成本修改（可选：校验和/签名），但不应当作为强反作弊手段

建议：

- `checksum`：对 `meta + data` 做哈希（如 SHA-256）用于损坏检测
- `signature`：可选（HMAC），主要用于“检测是否被改动”，不保证无法伪造

## 版本兼容性

### 版本号管理

- `schemaVersion`：只反映存档结构与字段语义变化
- `gameVersion`：游戏发布版本（用于展示/排查问题）

### 迁移逻辑（示例）

```ts
export type SaveEnvelope = {
  meta: { schemaVersion: number } & Record<string, unknown>;
  data: any;
};

export function migrateSave(envelope: SaveEnvelope, toSchemaVersion: number): SaveEnvelope {
  let current = envelope.meta.schemaVersion;
  let next = envelope;

  while (current < toSchemaVersion) {
    if (current === 1) {
      next = migrate_1_to_2(next);
      current = 2;
      continue;
    }
    throw new Error(`No migration path from schemaVersion=${current}`);
  }

  next.meta.schemaVersion = toSchemaVersion;
  return next;
}

function migrate_1_to_2(env: SaveEnvelope): SaveEnvelope {
  // 示例：新增字段、重命名字段等
  env.data.player = env.data.player ?? {};
  env.data.player.factionStanding = env.data.player.factionStanding ?? {
    galacticTradeUnion: 0,
    independentMerchants: 0,
    outerRimAlliance: 0,
    corporateFederation: 0,
  };
  return env;
}
```

## 存档操作 API（建议）

### 存储层（SaveStore）

```ts
export interface SaveMeta {
  schemaVersion: number;
  gameVersion: string;
  slotId: number;
  saveName: string;
  timestampMs: number;
  isAutosave: boolean;
  playTimeSeconds?: number;
  lastPlanet?: string;
  difficulty?: string;
  thumbnailDataUrl?: string;
}

export interface SaveEnvelope<TData = any> {
  meta: SaveMeta;
  data: TData;
  checksum?: string;
  signature?: string;
}

export interface SaveStore {
  get(slotId: number): Promise<SaveEnvelope | undefined>;
  put(slotId: number, envelope: SaveEnvelope): Promise<void>;
  delete(slotId: number): Promise<void>;
  listMeta(): Promise<SaveMeta[]>;
}
```

说明：

- `SaveStore` 的默认实现可基于 IndexedDB；测试/Node 环境可用内存实现

## 错误处理

### 常见错误与处理（Web 语境）

```
1. 存储空间不足（QuotaExceededError）
   - 提示用户清理浏览器站点数据
   - 尝试删除旧备份/缩略图
   - 降低自动存档频率或关闭缩略图

2. 浏览器禁用/限制存储（无痕模式/隐私设置/第三方环境）
   - 提示用户更换模式或允许站点存储
   - 降级为“仅会话内存档”（可选）

3. 数据损坏（校验失败/JSON 解析失败）
   - 自动回退到 backup
   - 提供“导出损坏存档用于反馈”按钮（可选）

4. 版本不兼容（缺少迁移路径）
   - 显示明确提示与风险说明
   - 保留原始存档，不做破坏性写入
```

## 性能优化

建议按出现瓶颈再逐步引入：

- 限频（throttle）：避免每次小变更都写入 IndexedDB
- 拆分数据：将大体积数据（如缩略图）独立存储，降低写放大
- 增量保存：dirty flags + 局部写入
