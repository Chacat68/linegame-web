## 二、技术决策

### 2.1 语言与构建

**维持 Vanilla JS + ES Module 路线，不引入构建工具**

理由：
- 现有原型已用 Vanilla JS 可运行，迁移 TypeScript 成本高且不是 MVP 瓶颈
- 浏览器已原生支持 ES Modules（`type="module"`），无需打包
- 后续如有必要可通过 `esbuild` 一步加入 TS，不影响现阶段迭代

**改造方向：** 将现有 4 个全局 IIFE 脚本拆分为 ES Module 文件，通过 `import/export` 组织依赖。

| 模式 | 用途 | 位置 |
|------|------|------|
| **事件总线（EventBus）** | 模块间解耦通信，如"交易完成"触发日志、成就、声望 | `core/EventBus.js` |
| **状态对象（GameState）** | 唯一数据源，所有系统读写统一对象 | `core/GameManager.js` |
| **命令模式** | 交易/升级操作可撤销/记录，便于存档 | `systems/trade/TradeSystem.js` |
| **观察者** | UI 订阅状态变更，避免全量重绘 | `ui/` 各模块 |

## 三、分阶段实施计划

### 第一阶段：代码重构（不加新功能）

**目标：** 在不改变可玩行为前提下，将现有 4 个文件拆分为 ES Module 结构，为后续扩展奠定基础。

**工作项：**

1. `index.html` — 将 `<script>` 标签改为 `type="module"` 加载 `js/main.js`
2. 拆分 `data.js` → `data/goods.js` + `data/systems.js` + `data/upgrades.js`
3. 拆分 `economy.js` → `systems/economy/Economy.js`（保留 WASM polyfill 注释）
4. 拆分 `renderer.js` → `ui/Renderer.js`
5. 拆分 `game.js` — 按职责分发：
   - 贸易逻辑 → `systems/trade/TradeSystem.js`
   - 飞船状态 → `systems/ship/ShipManager.js`
   - UI 渲染方法 → `ui/MarketUI.js` + `ui/ShipUI.js` + `ui/HUD.js`
   - 交互事件绑定 → `ui/MapUI.js` + `ui/Modal.js`
   - 主循环 → `core/GameManager.js`
6. 引入 `core/EventBus.js`，将"交易→日志"、"航行→日志"改为事件驱动

**验收：** 游戏原有，功能与重构前完全一致，浏览器控制台无报错。

**预估工作量：** 1 周

#### 3.1 供需驱动动态价格（`systems/economy/SupplyDemand.js`）

参考：`05_经济与市场系统.md`

当前实现：价格 = 基础价 × 星球系数 × 随机日修正  
目标实现：引入库存/供应/需求变量，供应量随玩家批量购买下降（即时收购影响同一天内后续价格）

```js
// 每个星系、每种商品维护：
{
  supply: 100,     // 当前库存（每日恢复部分）
  demand: 50,      // 当前需求量（NPC 消耗 + 随机事件影响）
  baseSupply: 100, // 星球类型决定的基准供应
}

// 价格公式（参考设计文档）：
price = basePrice × (demand / supply) × starSystemMultiplier × dayModifier
```

## 六、风险与约束

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| ES Module 重构引入回归 Bug | 中 | 重构期间每步验证游戏可玩 |
| IndexedDB API 在某些浏览器限制 | 低 | 降级到 localStorage 作为备选 |
| 供需模型导致数值失衡 | 高 | 参照 `数值平衡公式集.md` 做封顶保护，灰度测试 |
| 随机事件频率太高/低影响节奏 | 中 | A/B 配置，从 10% 概率起步调整 |
| 任务时限设计导致挫败感 | 中 | 宽松截止时间 + 失败任务给小额补偿 |

## 七、开发规范

1. **每个 JS 模块** 顶部注释声明：依赖模块、对外暴露 API、涉及的设计文档
2. **GameState 只读规则**：UI 层不直接修改 state，通过 `GameManager.dispatch(action)` 或系统方法操作
3. **价格计算** 保留 `calculatePrice` 函数签名（WASM polyfill 迁移路径）
4. **中文字面量** 统一放在 `data/` 目录的数据文件中，逻辑层不硬编码文案
5. **存档版本** 每次修改 GameState 结构时递增 `schemaVersion`，编写对应迁移函数
