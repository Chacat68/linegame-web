## 四、与旧文档的主要偏差

1. 仓库并非“仅文档未实现”，而是已具备可运行的多系统版本。  
2. 技术路径已落在 `Vanilla JS + ES Modules`，不是 `src/*.ts` 的未实现阶段。  
3. 存档方案与设计文档存在偏差：当前实现是 localStorage，不是 IndexedDB。  
4. 多数 P1/P2 功能（任务、成就、事件、派系、舰队）已提前落地，后续重点应转向“平衡与稳态”而非“从零开发”。

## 五、优先行动（建议执行顺序）

### P0（本周）

1. **统一文档口径**：把 `README.md`、`实现方案.md`、`MVP路线图.md` 中旧路径（如 `js/game.js`、`src/*.ts`）统一到当前结构。  
2. **冻结 GameState 与存档契约**：沉淀字段清单、默认值、`schemaVersion` 变更规则，避免后续破档。  
3. **存档策略定稿**：二选一：
   - A：短期保持 localStorage（低风险，快速迭代）
   - B：迁移 IndexedDB（更高容量，适合后续内容增长）

### P1（1-2 周）

1. **经济平衡参数化**：将供需回弹、峰值概率、税率影响抽为可调配置。  
2. **事件池分层**：按前中后期权重分桶，避免早期事件过重或后期重复感。  
3. **教程与任务联动校准**：减少新手阶段信息负担，强化首小时目标感。

### P2（2-4 周）

1. **最小自动化回归**：覆盖存档读写、价格计算、任务推进三类高风险逻辑。  
2. **性能与可维护性**：在 `GameManager` 引入更明确的 action 边界，降低模块耦合。  
3. **远期系统预留**：为音频、本地化、多人预留数据接口，不急于一次性实现。

---

## 六、结论

当前项目已跨过“原型是否可玩”的阶段，进入“设计口径统一 + 数值平衡 + 质量工程化”阶段。  
下一轮工作的核心，不是再补一个大系统，而是把已有系统打磨到稳定、可扩展、可维护。
